同一产品，不同角色的增长触发点

技术负责人

关心发布风险可控和门禁标准统一。核心交付是 `--fail-on` 策略 + CI 模板化接入包。

产品负责人

关心质量问题是否影响上线节奏。核心交付是 Fleet 趋势和跨团队风险优先级看板。

增长负责人

关心“能否复制传播”。核心交付是分享化页面、案例模板、可复用分发脚本。

30 / 60 / 90 执行节奏

每个阶段都对应产物、指标、扩散动作

30D

标准化接入

落地 `release/security` 双 preset，固定核心扫描参数，输出团队接入模板（CLI + Action 示例）。

60D

组织级可见性

推广 `fleet-scan` 与历史趋势，沉淀每周 Fleet Digest；用执行遥测追踪慢仓与失败仓恢复节奏。

90D

对外传播与裂变

发布案例页面和能力对照页，形成“可复制的治理故事”；通过社区渠道触发新增仓库接入。

增长型功能包

建议作为下一批 release 的 feature set

Pack 1: Onboarding Kit

`repos.txt` 导出 + `--repos-file` 批处理 + 预设参数模板， 把“首次接入成本”降到最低，支持一线团队自助扩展。 

node src/cli.js fleet-scan --discover-root ../workspace --discover-depth 4 --export-repos reports/repos.txt --export-only
node src/cli.js fleet-scan --repos-file reports/repos.txt --history-dir reports/fleet-history --format json --fail-on none

Pack 2: Sharing Kit

Fleet HTML + 宣传页并行发布，生成对外展示入口，把“扫描结果”变成“可传播产品资产”。 

node src/cli.js fleet-scan repo-a repo-b --history-dir reports/fleet-history --scan-out-dir reports/fleet-reports --scan-format html --format html --out reports/fleet.html
npx serve site
