30 / 60 / 90 执行节奏

每个阶段都有明确产物与指标

30D

标准化接入

固定 release / security preset 与默认阈值，输出团队接入模板（CLI + Action 示例）。

60D

组织级可见性

推广 fleet-scan 与历史趋势沉淀，开始每周 Fleet Digest，追踪失败仓与恢复时长。

90D

外部传播与扩散

发布能力页和案例页，把治理结果包装成可复用故事，驱动更多仓库接入。

增长功能包

建议作为下一阶段 release 的产品化清单

Pack 1: Onboarding Kit

`repos.txt` 导出、批量扫描入口与参数模板，目标是把“首次接入成本”压到最低，支持一线团队自助扩展。 

node src/cli.js fleet-scan --discover-root ../workspace --discover-depth 4 --export-repos reports/repos.txt --export-only
node src/cli.js fleet-scan --repos-file reports/repos.txt --history-dir reports/fleet-history --format json --fail-on none

Pack 2: Sharing Kit

Fleet HTML 与宣传页并行发布，把扫描结果升级为可传播资产，支撑内外部复盘、汇报和扩散。 

node src/cli.js fleet-scan repo-a repo-b --history-dir reports/fleet-history --scan-out-dir reports/fleet-reports --scan-format html --format html --out reports/fleet.html
npx serve site

上线与分发渠道

先预览，再自动化发布

GitHub Pages

适合 OSS 项目主站，仓库内已有 workflow 时可跟随 `site/` 变更自动发布。

报告分发

Fleet 报告链接可直接投放到周报、群聊和复盘文档，形成低成本传播入口。

模板扩散

通过参数模板和执行手册，把实践复制到更多项目与团队。

下一步做什么最有效？

优先上线 Onboarding Kit，再同步 Sharing Kit，先打通内循环再做外传播。
