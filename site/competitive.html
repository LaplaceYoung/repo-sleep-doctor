三步落地路径

01

单仓扫描

先拿到当前仓库的风险画像，快速识别 merge marker、潜在密钥、调试残留等发布阻塞项。

02

团队门禁

03

Fleet 治理

批量扫描多仓并追踪执行遥测，定位失败仓和恢复节奏，形成组织级治理闭环。

直接收益

标准化 CI 门禁，统一质量阈值，减少跨团队沟通摩擦与重复修复成本。

推荐命令

node src/cli.js fleet-scan --discover-root ../workspace --discover-depth 4 --export-repos reports/repos.txt --export-only
node src/cli.js fleet-scan --repos-file reports/repos.txt --history-dir reports/fleet-history --format html --out reports/fleet.html --continue-on-error --fail-on p1

直接收益

统一多仓治理视角，快速识别失败仓、慢仓、风险热点和恢复节奏，支撑管理决策。

FAQ 与执行建议

常见问题直接给结论

Q1: 首次接入建议从哪个阈值开始？

先用 `--fail-on none` 建立基线，再逐步收敛到 `p1`；团队成熟后再考虑 `p0` 或更严策略。

Q2: 多仓场景如何降低路径维护成本？

使用 `--discover-root` + `--export-repos` 生成 `repos.txt`，将“仓库发现”和“仓库扫描”解耦。

Q3: 如何向管理层展示治理效果？

固定输出 Fleet HTML 和 execution log，每周追踪失败仓数量、恢复时长与热点规则趋势。

要先在哪个场景试点？

建议先从“团队 CI”开始，再扩展到 Fleet 治理。
